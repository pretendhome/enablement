# Program Architecture
**Purpose**: Define the full enablement program as a system, not just a single loop.

## If I Were Starting Over

I would not begin with a folder of references.
I would begin with the experience.

The system would have five layers from day one:

1. **Orientation**
What mode are we in?
Examples:
- interview prep
- onboarding
- product mastery
- demo readiness
- executive briefing

2. **Reference gating**
Only the minimum relevant reference is exposed before a question.

3. **Retrieval and judgment**
Answer live, get scored, refine, repeat.

4. **Pressure testing**
Shorter answer, skeptical answer, technical answer, executive answer.

5. **Artifact performance**
For any deliverable that must be recorded or presented, use a separate loop for pacing, delivery, and failure recovery.

## The Program Model

### Mode A: Learn

Goal:
- understand a concept well enough to explain it

Output:
- one clean answer

### Mode B: Retrieve

Goal:
- answer from memory under time pressure

Output:
- pass / refine / reset

### Mode C: Pressure

Goal:
- survive hostile follow-up and variant framing

Output:
- resilient answer, not just rehearsed answer

### Mode D: Perform

Goal:
- deliver the answer live or on video

Output:
- usable artifact

### Mode E: Debrief

Goal:
- capture what failed, what passed, and what to fix next

Output:
- next-session state

## Why This Is Better

The first version of the system correctly emphasized judged retrieval.
What it lacked was enough explicit structure around:
- session continuity
- performance artifacts
- failure recovery
- progression from concept to recording

This architecture fixes that.

## Design Rules

1. **Read late**
References are support, not the experience.

2. **Answer early**
You learn more from a failed answer than from another 20 minutes of reading.

3. **Refine narrowly**
Do not rewrite the whole answer when one weak point is the problem.

4. **Track state**
A strong enablement system remembers what already passed and what still breaks trust.

5. **Separate explanation from performance**
A person can understand the content and still fail the recording.
The system must train both.
