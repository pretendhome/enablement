# Iteration 7: Generalization

**Date**: 2026-03-16
**Focus**: Making the enablement system work for any role in any domain, not just "enable someone on agentic tools"
**Problem it solves**: The current system is implicitly "help a non-technical person build an AI software suite." But the architecture — lens, brief, coaching loop, progress file — has nothing inherently agentic about it. If you squint, it's a general enablement engine with one domain pack loaded. This iteration makes that explicit: extract the reusable engine from the domain-specific content.
**Lineage**: Codex CONTENT_LAYERING.md (three-layer separation), Iteration 3's Convergence Brief (the 7 stages to generalize), Gap-007 from Iteration 3 (worked examples missing).

---

## The Key Insight

The 7 stages from Iteration 3 look domain-specific:

1. Foundations → First Instructions → Memory → Verification → Organization → Building → Autonomy

But the progression pattern underneath is universal:

1. **Orient** — Understand what you're building and why it matters to you
2. **First Use** — Do the simplest version of the core skill
3. **Retain** — Make what you learned persist across sessions
4. **Verify** — Build a habit of checking that your work is correct
5. **Organize** — Structure and protect what you've built
6. **Extend** — Build new capabilities on the foundation
7. **Own** — Maintain and grow independently

"First Instructions" is Orient → First Use for agentic tools. "Run your first query" is Orient → First Use for product analytics. "Create your first automated follow-up" is Orient → First Use for CRM automation. The stage names change. The progression doesn't.

## Content Layering Applied

Following Codex's CONTENT_LAYERING.md pattern — separate the engine from the content.

### Layer 1: Reusable Engine (Stable)

These components work for any domain without modification:

| Component | Why it's universal |
|---|---|
| **LearnerLens schema** | Identity, assessment, goals, state — applies to any learner |
| **Intake protocol** (5 questions) | "What do you spend your days doing?" works for anyone |
| **7-stage progression pattern** | Orient → First Use → Retain → Verify → Organize → Extend → Own |
| **Coaching loop** (5 steps) | Resume → Do → Check → Capture → Advance |
| **Interaction patterns** (4) | Stuck, skip, off-script, overwhelmed |
| **Progress file format** | About Me, Where I Am, What's Working, What I've Built, Session Log |
| **Session handoff protocol** | Paste at start, generate at end |
| **Verification patterns** (5) | Show-me, teach-back, outcome check, before-and-after, trust-and-verify |
| **Safety principles** (4) | Vendor uncertainty, backup, control distinction, one-way doors |
| **Language rules** (6) | Plain language first, metaphors, show the problem, one concept, their words, never "it's simple" |
| **Brief generation rules** (6) | Skip passed stages, adapt to tools, pace to budget, name gaps, link to lens, living document |

**Nothing in Layer 1 references "AI," "agentic," "instructions," "steering files," or any domain concept.** It's pure enablement infrastructure.

### Layer 2: Domain Pack (Changes by context)

A domain pack is a self-contained document that plugs into the engine for a specific enablement context. It contains:

1. **Domain definition** — What the learner is being enabled on, in one sentence
2. **Stage instantiation** — What each of the 7 universal stages means in this domain (names, framing, activities, success criteria, time estimates)
3. **Translation table** — Domain-specific terms and their plain-language equivalents
4. **Worked examples** — 2-3 example briefs for different learner profiles in this domain
5. **Tool landscape** — The typical tools a learner in this domain will encounter

A domain pack does NOT contain the coaching loop, the lens, the verification patterns, or the safety principles — those belong to the engine.

### Layer 3: Learner Instance (Changes fastest)

This is the individual learner's state:

- Their specific LearnerLens
- Their progress file
- Their session history
- Their specific tool configurations

Layer 3 is generated by the engine + domain pack together, and evolves with every session.

## The Universal Stage Pattern

Each stage in the progression has a universal structure. The domain pack fills in the specifics.

```
Stage: [Universal name]
Domain name: [What this stage is called in this domain]
Framing: [Plain-language explanation for the learner — domain-specific]
Activities: [1-3 concrete activities in the learner's tools — domain-specific]
Success criteria: [What "done" looks like — domain-specific]
Typical duration: [Time estimate — domain-specific]
```

### Stage definitions:

**1. Orient**
- Purpose: The learner understands what they're building and why it matters to their specific work
- Universal success criterion: The learner can explain, in their own words, what they're building and why
- Typical pattern: Review current state → identify one pain point → see a worked example of what's possible

**2. First Use**
- Purpose: The learner does the simplest version of the core skill, producing a real artifact
- Universal success criterion: The learner has created one working artifact they can point to
- Typical pattern: Build something small together → test it → see the improvement

**3. Retain**
- Purpose: What the learner built persists across sessions — it doesn't disappear when they close the tool
- Universal success criterion: The learner reopens their tool and finds their work intact, or knows exactly where to find it
- Typical pattern: Experience the problem (it's gone!) → create the persistence mechanism → verify it survived

**4. Verify**
- Purpose: The learner builds a habit of checking that their system is working correctly
- Universal success criterion: The learner has done at least one check-in and corrected something
- Typical pattern: Run a check → find something wrong → fix it → establish a cadence

**5. Organize**
- Purpose: The learner's work is structured, findable, and backed up
- Universal success criterion: Everything has a place, the learner knows where to find it, and it's backed up
- Typical pattern: Audit → organize → back up → create a safety net

**6. Extend**
- Purpose: The learner builds new capabilities on the foundation — not just more of the same, but genuinely new functionality
- Universal success criterion: The learner has built something new that works reliably
- Typical pattern: Identify a need → design it → build it → test it → integrate it

**7. Own**
- Purpose: The learner can maintain and extend their system independently
- Universal success criterion: The learner builds something new without guidance and can explain their system to someone else
- Typical pattern: Build solo → explain to others → create a restart document

## Example Domain Packs

### Domain Pack A: Agentic Enablement (Current System)

**Domain definition**: Enable a non-technical professional to build their own AI software suite — custom instructions, memory, verification, and specialized assistants.

| Universal Stage | Domain Name | Key Activity |
|---|---|---|
| Orient | Foundations | Review current AI use, identify one task AI could do better if it "knew" you |
| First Use | First Instructions | Write a plain-language instruction set for your AI, test it |
| Retain | Memory | Create a memory note your AI reads at conversation start |
| Verify | Verification | Check: is my AI still following instructions? Is the memory accurate? |
| Organize | Organization | Structure instructions, memory, backups into a clean system |
| Extend | Building | Build a specialized assistant or automated workflow |
| Own | Autonomy | Build a new capability solo, explain your system to a colleague |

**Translation table**: Steering file, memory file, verification loop, context window, convergence, one-way door, agent, progress file, portable, persistence (see PROMPT.md).

---

### Domain Pack B: Product Analytics

**Domain definition**: Enable a product manager to make data-driven decisions using analytics tools — building dashboards, running queries, interpreting metrics, and establishing a measurement culture on their team.

| Universal Stage | Domain Name | Key Activity |
|---|---|---|
| Orient | Why Data Matters | Review current decisions made by gut, identify one that would benefit from data |
| First Use | First Dashboard | Build one dashboard with one metric that answers a question they actually have |
| Retain | Saved Queries | Save queries and dashboards so they survive across sessions and are findable |
| Verify | Metric Hygiene | Check: is this metric measuring what I think? Is the data source reliable? |
| Organize | Analytics Home | Organize dashboards by purpose, set up alerts for critical metrics |
| Extend | Custom Analysis | Build a segment analysis, a funnel report, or a cohort comparison |
| Own | Data Culture | Create a new report independently, teach a teammate to read a dashboard |

**Translation table**:
- "Query" → "A question you ask your data — written in a way the tool understands"
- "Dashboard" → "A page that shows your key numbers, updated automatically"
- "Metric" → "A number that tells you how something is going"
- "Funnel" → "The steps people take from first visit to the thing you want them to do"
- "Cohort" → "A group of people who did the same thing at the same time"
- "Segment" → "A slice of your users — like 'people who signed up last month' or 'users in Europe'"

**Worked example**: A PM at a 50-person SaaS company. Uses Mixpanel and Google Sheets. Has 30 minutes twice a week. Comfort level: 1 (looks at dashboards others built but never created one). Near-term goal: "Know which features are actually used vs. which ones we built and nobody touches."

Brief would start at Stage 1 (Orient) — identify one feature they're unsure about, look at usage data together. Stage 2 (First Dashboard) — build a feature adoption dashboard in Mixpanel showing daily active users per feature. By Stage 4 (Metric Hygiene), they're checking whether "active users" counts page loads or actual interactions, and correcting the definition.

### Domain Pack C: CRM Automation

**Domain definition**: Enable a sales rep to automate repetitive pipeline work — follow-ups, lead scoring, task routing — so they spend more time selling and less time on admin.

| Universal Stage | Domain Name | Key Activity |
|---|---|---|
| Orient | Where Your Time Goes | Map a typical week — how much is selling vs. admin? Identify one admin task to automate |
| First Use | First Automation | Create one automated follow-up email that triggers after a demo |
| Retain | Template Library | Save email templates and automation rules so they persist across deals |
| Verify | Automation Audit | Check: are the right emails going to the right leads? Any misfires? |
| Organize | Pipeline Hygiene | Organize automations by stage, clean up dead rules, tag templates |
| Extend | Multi-Step Sequences | Build a sequence: demo → follow-up → if no response → second follow-up → if response → route to closer |
| Own | Sales System Owner | Create a new automation for a different stage, train a new rep on the system |

**Translation table**:
- "Automation" → "A rule that does something for you automatically — like 'when X happens, do Y'"
- "Sequence" → "A chain of steps that happen one after another, with waits in between"
- "Trigger" → "The event that starts an automation — like 'lead replies' or '3 days pass'"
- "Lead scoring" → "A way to rank which leads are worth your time right now"
- "Pipeline" → "All your deals, organized by how close they are to closing"
- "Template" → "A pre-written email you can reuse and customize for each lead"

**Worked example**: A sales rep at a B2B SaaS company. Uses HubSpot and Gmail. Has 20 minutes after Friday pipeline review. Comfort level: 1 (uses CRM to log calls but no automations). Near-term goal: "Stop forgetting to follow up after demos."

Brief would start at Stage 1 (Orient) — count follow-ups missed last month, calculate the revenue at risk. Stage 2 (First Automation) — create a HubSpot workflow that sends a template email 24 hours after a deal moves to "Demo Complete." By Stage 4 (Automation Audit), they're checking that the template sounds right and no automation fired for a deal that was already closed-won.

## Creating a New Domain Pack

To add support for a new domain:

1. **Write a one-sentence domain definition.** What is the learner being enabled on?

2. **Instantiate the 7 stages.** For each universal stage (Orient → First Use → Retain → Verify → Organize → Extend → Own), define:
   - A domain-specific name
   - A plain-language framing paragraph
   - 1-3 activities in the tools a learner in this domain would actually use
   - Clear success criteria
   - A time estimate

3. **Build the translation table.** List every domain-specific term the system might need to use, with a plain-language equivalent. Follow the format from Iteration 2's Language Calibration.

4. **Write 1-2 worked examples.** Pick a realistic learner persona — role, tools, time budget, comfort level, goal. Walk through how the brief would look for them. This makes the domain pack concrete and testable.

5. **Identify the tool landscape.** What tools do learners in this domain commonly use? This helps the system adapt activities to the learner's actual tools during brief generation.

A domain pack should fit in a single document. It's scenario content, not a codebase.

## Prompt Section (for assembly into PROMPT.md)

The generalization iteration does not add a new section to PROMPT.md. Instead, it reframes how the existing sections are understood:

- Sections 1-4 and Section 5 are the **engine** (Layer 1). They work for any domain.
- The specific stage names, activities, and translation table entries currently in the prompt are the **default domain pack** (Agentic Enablement).
- When deployed for a different domain, the engine stays the same and the domain pack swaps.

No prompt changes needed — the current PROMPT.md already works as the agentic enablement instance. The generalization is an architectural understanding, not a prompt rewrite. The domain pack structure is documented in this iteration for anyone who wants to create a new one.

However, one addition to the prompt is warranted — a note in the "Who You Are" section that makes the engine's generality explicit:

```markdown
### Domain Awareness

The stages, activities, and terminology in this prompt are configured for a specific domain.
The underlying engine — the lens, the coaching loop, the progress file, the verification
patterns — works for any domain. If you are loaded with a different domain pack,
follow that pack's stage definitions while using the same engine.
```

## Gaps

1. **No domain pack registry.** As more domain packs are created, they need a lightweight index — which packs exist, who created them, when they were last tested. Not needed now (only one pack exists), but will matter at scale.

2. **Cross-domain enablement.** What happens when a learner needs two domain packs simultaneously — e.g., a PM who needs both product analytics AND agentic tool enablement? The system currently assumes one domain per learner. A future iteration should define how to compose packs.

3. **Domain pack quality standards.** Not all domain packs will be equally good. A poorly written pack with vague success criteria will produce bad briefs. Needs a lightweight review checklist — similar to the quality checks in Iteration 1 for the lens.

4. **The "menu of possibilities" is still missing.** Iteration 3's Gap-004 (the "I don't know what I don't know" problem) is partially addressed by worked examples in each domain pack. But the Orient stage still needs a curated set of outcomes other learners have achieved. This is domain-specific content that each pack should include.
